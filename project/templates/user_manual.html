数据导入指南

本部分详细说明了数据导入的规则、字段限制和枚举值要求,是保证数据质量的关键。

核心导入原则
  • 使用最新模板: 务必从系统中下载最新的Excel导入模板,不要使用旧版或自行修改表头。
  • 枚举值精确匹配: 所有涉及选择的字段(如采购方式),必须填写系统预定义的标准枚举值,不能有任何偏差。
  • 关联数据先行: 导入有关联关系的数据时(如合同关联采购),请确保被关联的数据已存在于系统中。
  • 编号唯一性: 各类编号(项目编号、采购编号、合同编号)是唯一标识,不能重复。
  • 日期格式: 所有日期字段必须使用 YYYY-MM-DD 格式,例如 2025-10-29。

关键业务规则
  • 规则1: 补充协议和解除协议必须关联主合同,系统会自动继承主合同的来源类型和采购关联
  • 规则2: 主合同不能关联其他合同
  • 规则3: 框架协议可作为多个具体合同的基础
  • 规则4: 当"合同来源"为"采购合同"时,必须关联采购项目(填写关联采购编号)

  • 规则8: 合同金额必须大于0,单位为"元",支持最多2位小数
数据导入顺序建议
  1. 先导入项目数据(如果合同需要关联项目)
  2. 再导入采购数据(如果是采购合同)
  3. 然后导入主合同数据
  4. 最后导入补充协议和解除协议
常见错误及解决方法
  • 错误: "补充协议必须关联主合同"
    解决: 确保已填写"关联主合同编号",且该主合同已存在于系统中
  • 错误: "采购合同必须关联采购项目"
    解决: 当"合同来源"为"采购合同"时,必须填写"关联采购编号"
  • 错误: "直接签订合同不应关联采购项目"
    解决: 当"合同来源"为"直接签订"时,清空"关联采购编号"字段
  • 错误: "主合同不能关联其他合同"
    解决: 如果"文件定位"为"主合同",清空"关联主合同编号"字段
  • 错误: 合同编号包含特殊字符
    解决: 移除编号中的 / \ ? # 等特殊字符,可使用下划线或连字符替代

履约评价数据导入规范

供应商履约评价导入模板支持宽表和长表两种格式。

导入模板格式说明
长表格式（灵活动态）

履约评价导入采用长表格式，支持动态年份和次数，无需预先定义列。每个评价记录占一行，年度评价和不定期评价通过"评价维度"字段区分。

格式特点：
  • 灵活扩展：支持任意年份和次数，无需修改模板结构
  • 动态字段：通过"评价维度"字段标识是年度评价还是不定期评价
  • 简单明了：每行一个评价记录，字段固定，易于理解和填写
字段限制与说明
字段名必填类型限制与说明
评价编号否字符串唯一标识。留空则自动生成格式：PJ+合同编号（如：PJHT2025001），也可自定义
合同序号是字符串必须为系统中已存在的合同序号（关联到具体合同）
供应商名称否字符串可自动从关联合同获取，无需手动填写
评价类型否枚举必须为"末次评价"或"履约过程评价"
末次评价得分否数字满分100分，权重60%
履约综合评价得分否数字可直接导入，或系统根据末次和过程评价自动计算
评价维度条件必填字符串用于区分评价类型，格式：年度_YYYY（如：年度_2023）或 不定期_N（如：不定期_1）
评价得分否数字该维度的评价得分，满分100分
综合评分计算规则
自动计算公式

当CSV未提供"履约综合评价得分"时，系统自动计算：

综合评分 = 末次评价得分 × 60% + 过程评价平均分 × 40%

过程评价包括所有年度评价和不定期评价的平均值。

评分等级：

  • 90分及以上：优秀
  • 80-89分：良好
  • 70-79分：合格
  • 70分以下：不合格

合同管理模块

合同管理模块支持主合同、补充协议、解除协议和框架协议的全生命周期管理。

核心功能
1. 合同类型管理
  • 主合同：独立存在的完整合同，可关联采购项目或直接签订
  • 补充协议：必须关联主合同，用于调整合同金额、工期等条款
  • 解除协议：必须关联主合同，用于终止合同
  • 框架协议：框架性质的合同，可作为多个具体合同的基础
2. 合同来源分类
  • 采购合同：来源于采购项目，必须关联采购编号
  • 直接签订：不经过采购流程直接签订的合同，不能关联采购
3. 关联关系
  • 合同可关联到具体项目（可选）
  • 补充协议和解除协议自动继承主合同的来源类型和采购关联
  • 系统自动追踪主合同及其所有补充协议的总金额
关键字段说明
字段说明备注
合同编号主键，唯一标识不能包含 / \ ? # 等特殊字符，用于生成履约评价编号
合同序号合同的序号支持字符串格式，如 BHHY-NH-001，用于履约评价关联
文件定位区分合同类型主合同/补充协议/解除协议/框架协议
关联主合同补充协议/解除协议必填建立合同间的层级关系
含税签约合同价签订时的合同金额单位：元，最多2位小数
合同签订日期正式签署日期用于归档监控的起算时间
合同工期/服务期限履约期限文本格式，可描述具体日期区间
履约担保退回时间担保金退回日期用于跟踪担保金管理
业务规则
重要规则
  • 补充协议必须关联主合同，系统会自动继承主合同的来源类型
  • 采购合同必须关联采购项目，直接签订合同不能关联采购
  • 付款进度计算会考虑主合同+所有补充协议的总金额
  • 如果有结算价，付款比例以结算价为基准；否则以合同价+补充协议为基准
  • 合同归档时限：签订后30天内必须归档
统计与分析

系统自动提供以下统计分析功能：

  • 合同总览：按年份、项目、类型统计合同数量和金额
  • 付款进度：实时跟踪每个合同的付款比例和剩余待付金额
  • 归档监控：自动识别超期未归档的合同，按严重程度分级预警
  • 关联完整性：检查补充协议、采购合同的关联关系是否正确
  • 履约评价：每个合同可关联履约评价记录，评价编号自动使用"PJ+合同编号"格式

履约评价模块

供应商履约评价模块支持末次评价、过程评价（年度+不定期）的综合管理和自动计分。

核心功能
1. 评价编号管理
  • 自动生成：创建评价时，如不填写编号，系统自动生成格式：PJ+合同编号
  • 自定义支持：用户可在表单中手动输入自定义编号
  • 唯一性保证：评价编号作为主键，确保每条评价记录唯一标识
2. 评价体系
  • 末次评价：合同履约完成后的最终评价，权重60%
  • 过程评价：包括年度评价和不定期评价，权重40%
  • 综合评分：系统自动计算或手动导入
3. 动态扩展能力
  • 支持任意年份的年度评价（如2019年、2020年、2023年...）
  • 支持任意次数的不定期评价（第1次、第2次...）
  • 无需修改数据库结构，通过JSONField灵活存储
4. 供应商约谈记录
  • 约谈类型：违约约谈、履约沟通、商务洽谈、投标前沟通
  • 状态跟踪：待整改、整改中、已完成、仅记录
  • 关联管理：可关联具体合同或独立记录
评分计算规则
综合评分自动计算

综合评分 = 末次评价得分 × 0.6 + 过程评价平均分 × 0.4

过程评价计算

过程评价平均分 = (所有年度评价得分之和 + 所有不定期评价得分之和) / 评价总次数

示例：
某供应商评价数据：
• 末次评价：90分
• 2019年度：85分
• 2020年度：88分
• 不定期评价第1次：92分

过程评价平均 = (85 + 88 + 92) / 3 = 88.33分
综合评分 = 90 × 0.6 + 88.33 × 0.4 = 54 + 35.33 = 89.33分
供应商约谈管理
约谈类型
  • 违约约谈：供应商违反合同条款时的正式约谈
  • 履约沟通：日常履约过程中的沟通协调
  • 商务洽谈：商务条款、付款等事项的洽谈
  • 投标前沟通：投标前的技术或商务交流
跟进状态
  • 待整改：问题已提出，供应商需要采取行动
  • 整改中：供应商正在进行整改
  • 已完成：问题已解决，整改完成
  • 仅记录：无需整改，仅作记录
记录要点

约谈记录应包含：

  • 约谈日期和参与人员
  • 约谈原因和详细内容
  • 提出的整改要求（如适用）
  • 处理结果和后续跟进情况
评价编号规则
编号生成规则
  • 自动生成：系统默认使用"PJ+合同编号"格式，例如合同编号为HT2025001，评价编号自动生成为PJHT2025001
  • 自定义编号：用户可在创建或导入时自定义评价编号，只要保证唯一性即可
  • 关联依据：履约评价通过"合同序号"字段关联到具体合同，而非通过评价编号
  • 唯一性保证：评价编号是主键，系统自动确保不重复
应用场景
1. 供应商绩效评估

系统根据综合评分自动生成供应商绩效等级，用于：

  • 供应商优选和淘汰
  • 采购决策参考
  • 续签合同评估
2. 问题供应商管理

通过约谈记录追踪：

  • 多次违约的供应商
  • 需要重点关注的履约问题
  • 整改落实情况
3. 统计分析

支持按供应商、合同、评分等级进行统计分析，为供应商管理决策提供数据支撑。

统计分析模块计算逻辑

本部分详细解释统计分析页面 (monitoring/statistics.html) 中各项数据的计算方法,源于 project/services/statistics.py。

核心指标卡数据计算详解

统计分析页面的顶部卡片展示了最关键的业务指标,以下是每个指标的详细计算方法:

采购统计核心指标

采购项目总数 (Total Count)

对所有符合筛选条件的采购记录,按"采购项目名称"进行去重后的项目总数。

逻辑:COUNT(DISTINCT project_name)

总中标金额 (Total Winning Amount)

对所有采购记录按"采购项目名称"去重,对每个项目名称取其所有记录中最大的"中标金额",然后将这些最大值求和。

逻辑:SUM(MAX(winning_amount) GROUP BY project_name)

资金节约率 (Savings Rate)

基于去重后的数据计算节约率。

公式:(去重后总预算金额 - 去重后总中标金额) / 去重后总预算金额 * 100%

平均采购周期 (Avg Cycle Days)

计算从"采购需求书审批完成日期"到"中标通知书发放日期"的平均天数。此项不进行去重。

公式:AVERAGE(notice_issue_date - requirement_approval_date)

合同统计核心指标

合同总数 (Total Count)

所有符合筛选条件的合同记录总数。

合同总金额 (Total Amount)

所有符合筛选条件的合同记录的 `contract_amount` 字段之和。

逻辑:SUM(contract_amount)

付款与结算核心指标

付款总金额 (Total Payment Amount)

所有符合筛选条件的付款记录的 `payment_amount` 字段之和。

预计剩余支付 (Estimated Remaining)

所有主合同的"基准金额"之和减去其"已付总额"之和。基准金额优先取结算价,若无则取(主合同金额+补充协议金额)。

结算率 (Settlement Rate)

已办理结算的主合同数量占所有主合同总数的比例。

公式:(已结算主合同数 / 总主合同数) * 100%

采购统计 (get_procurement_statistics)

核心去重规则

关键逻辑: 为了避免因同一采购项目分多次导入或存在多条记录而导致的重复计算,采购统计的核心是按 采购项目名称 (`project_name`) 进行去重。

如果多个采购记录有相同的 `project_name`,系统会将其视为一个整体项目,并按以下规则合并统计:

  • 数量统计: 多个同名项目只计为 1 个。
  • 金额合并规则:
    • `预算金额`:取所有同名记录中的 最大值 (MAX)。
    • `中标金额`:取所有同名记录中的 最大值 (MAX)。
    • `采购控制价`:取所有同名记录中的 最大值 (MAX)。
  • 归档日期合并规则: 取所有同名记录中最新的一个非空日期。

此去重逻辑的目的是为了从多条可能重复或分阶段的记录中,提取出最能代表该采购项目整体情况的数值。该逻辑定义在 docs/采购统计去重说明.md 和 project/services/statistics.py 中。

关键指标计算公式

节约率 (Savings Rate) = (总预算金额 - 总中标金额) / 总预算金额 * 100%
注:此处的总金额均基于上述去重规则合并后的数据进行计算。
平均采购周期 (Avg Cycle Days) = AVERAGE(中标通知书发放日期 - 采购需求书审批完成日期)
注:此项计算 不进行去重,而是统计所有同时具有这两个日期的原始记录,以反映实际操作的平均耗时。
采购方式分布 (Method Distribution) = 对去重合并后的记录,按 `procurement_method` 字段进行分组计数和金额汇总。
合同与付款统计

合同统计 (get_contract_statistics)

合同统计相对直接,主要基于筛选条件(年份、项目)下的合同记录进行聚合。

总金额 (Total Amount) = SUM(所有符合条件合同的 'contract_amount')

类型/来源分布: 按 `file_positioning` (主合同, 补充协议等) 或 `contract_source` (采购合同, 直接签订) 字段分组,分别统计数量和金额。

付款统计 (get_payment_statistics)

预计剩余支付 (Estimated Remaining) = (所有主合同的基准金额之和) - (所有主合同的已付金额之和)

其中, 基准金额 的确定逻辑至关重要:

  1. 系统首先找到所有符合筛选条件的主合同。
  2. 对每一个主合同,检查其是否有对应的结算记录。
  3. 如果有结算价: 基准金额 = 结算价 (settlement.final_amount)。
  4. 如果没有结算价: 基准金额 = 主合同金额 + 所有其补充协议的金额之和。

这个逻辑确保了付款进度的计算是基于最准确的合同最终价值。

结算统计 (get_settlement_statistics)

核心去重规则

关键逻辑: 结算金额 (`settlement_amount`) 是针对整个合同的最终价值,但它可能记录在同一合同的多笔付款记录中。为避免重复计算结算总额,统计时按 主合同 进行去重。

系统通过以下步骤计算结算总额:

  1. 找出所有 `is_settled=True` 且 `settlement_amount` 不为空的付款记录。
  2. 根据这些付款记录找到其关联的合同,并最终追溯到主合同。
  3. 为每个唯一的主合同,只取一次 `settlement_amount` 值进行累加。
结算率 (Settlement Rate) = (已结算的主合同数量 / 总主合同数量) * 100%
注:总主合同数量会根据页面筛选条件(年份、项目)进行相应筛选。
结算差异分析 (Variance Analysis) = 结算金额 - (主合同金额 + 补充协议总金额)
此分析用于找出结算价与原始合同价差异最大的项目。

归档监控计算逻辑

源于 project/services/archive_monitor.py。

采购归档
  • 统计对象: 所有已填写 `结果公示发布时间` 的采购记录。
  • 归档时限: 40天。
  • 逾期判断: (当前日期 - 结果公示发布时间) > 40天 且 `归档日期` 为空。
  • 及时归档判断: (归档日期 - 结果公示发布时间) <= 40天。

预警分级 (按逾期天数)

  • 严重逾期 (红色): > 30天 (即公示后70天仍未归档)。
  • 中度逾期 (橙色): 16-30天 (即公示后56-70天仍未归档)。
  • 轻微逾期 (黄色): 1-15天 (即公示后41-55天仍未归档)。
合同归档
  • 统计对象: 所有已填写 `签订日期` 的主合同 (`file_positioning` 为 '主合同')。
  • 归档时限: 30天。
  • 逾期判断: (当前日期 - 签订日期) > 30天 且 `归档日期` 为空。
  • 及时归档判断: (归档日期 - 签订日期) <= 30天。

预警分级 (按逾期天数)

  • 严重逾期 (红色): > 30天 (即签订后60天仍未归档)。
  • 中度逾期 (橙色): 16-30天 (即签订后46-60天仍未归档)。
  • 轻微逾期 (黄色): 1-15天 (即签订后31-45天仍未归档)。

齐全性监控计算逻辑

源于 project/services/completeness.py。

字段齐全性

本模块检查核心业务记录的关键字段是否填写完整。齐全率 = (已填写的字段数 / 应填写的总字段数) * 100%。

采购记录检查 (共26个关键字段): 包括 `招采编号`, `采购项目名称`, `采购单位`, `中标单位`, `采购方式`, `预算金额`, `中标金额`, `结果公示发布时间`, `中标通知书发放日期` 等。

合同记录检查 (共17个关键字段): 包括 `合同序号`, `合同编号`, `合同名称`, `甲方`, `乙方`, `合同金额`, `签订日期`, `文件定位`, `合同来源` 等。

关联完整性

本模块检查数据之间的逻辑关系是否正确,并按严重性分级。

  • 【错误】补充协议/解除协议未关联主合同。
  • 【错误】采购合同未关联采购项目。
  • 【错误】付款总额超过合同金额(或结算价)。
  • 【警告】直接签订的合同错误地关联了采购项目。
  • 【警告】结算金额与合同金额差异超过10%。

更新监控计算逻辑

源于 project/services/update_monitor.py。

数据更新及时性

本模块监控核心业务事件发生后,其信息是否被及时录入或更新到系统中。

监控事件触发点

  • 采购: `结果公示发布时间`
  • 合同: `签订日期`
  • 付款: `付款日期`
  • 结算: `结算完成日期`

及时性判断标准

业务事件发生后,必须在 次月月底前 在系统中完成数据录入/更新。

判断依据: 系统会比较记录的最后更新时间 (`updated_at`) 与事件发生日期的"次月最后一天"这个截止日期。

示例: 一笔采购的"结果公示发布时间"是 2025年10月15日。那么,这条记录的系统更新时间 (`updated_at`) 必须早于或等于 2025年11月30日,才被认为是及时更新。
